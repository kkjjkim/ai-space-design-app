# AGENTS.md

> AI 코딩 도구용 설계도 + 안전장치. 깃 루트에 `AGENTS.md`로 저장.
> (클로드코드는 `CLAUDE.md`도 읽음. 둘 다 두면 같은 내용으로.)
> 사람이 보는 기획서는 `PRD.md` 참고.

## 1. 이 프로젝트가 뭔지
- 한 줄 설명: 고급 매장·레스토랑·대형카페를 준비하는 사장에게 "국가공인 경영지도사 + 디자인드비가 창업 첫걸음부터 매출까지 함께한다"를 보여주고, 무료 상담 신청(고객 DB)을 모으는 사이트.
- 핵심 행동: **무료 컨셉 상담 신청 폼 제출.** 모든 화면은 이 행동 하나를 위해 존재한다.
- 화면: 메인(원페이지 스크롤) / 포트폴리오 / 회사·신뢰 / 서비스 / 신청 완료.
- 사람이 보는 기획서·카피·디자인 방향은 `PRD.md`에 있다. **카피는 PRD 문구를 그대로 쓴다.**

## 2. 기술 스택 (이대로 간다. 이유 없이 바꾸지 마라)
- 프레임워크: Next.js (App Router)
- 스타일: Tailwind CSS + shadcn/ui
- DB·저장: Supabase (leads 테이블)
- 메일 알림: 신청 시 운영자에게 발송 (예: Resend). 키는 환경변수.
- 배포: Vercel
- 언어: TypeScript

## 3. 환경 / 명령어
- 패키지 매니저: pnpm (고정)
- Node: 20.x
- 설치: `pnpm install`
- 개발: `pnpm dev`
- 빌드: `pnpm build`
- 타입체크: `pnpm typecheck` (또는 `tsc --noEmit`)
- 린트: `pnpm lint`

## 4. 작업이 "끝났다"의 기준 (검증 루프 — 꼭 지켜라)
끝내기 전에 **반드시** 통과시켜라.
1. 타입체크 통과
2. 린트 통과
3. 빌드 성공
하나라도 깨지면 스스로 고친 뒤 다시 검사하고, 통과한 뒤에만 끝났다고 말해라.
"아마 될 것"이라고 넘기지 마라. 직접 명령어를 돌려 확인해라.

## 5. 작업 방식 (긴 작업에서 무너지지 않게)
- 큰 기능은 한 번에 다 만들지 마라. **먼저 계획을 단계로 나눠 보여주고** 동의받은 뒤 진행.
- 한 번에 한 가지만. 권장 순서: (1) 프로젝트 셋업·디자인 토큰 → (2) 공통 레이아웃·내비·고정 CTA → (3) 메인 섹션 1~8 → (4) 서브 페이지 → (5) 신청 폼 + Supabase 저장 + 메일 알림 → (6) 반응형·성능·점검.
- 진행 상황을 `PROGRESS.md`에 짧게 기록(끝낸 것/다음 할 것).
- 막히면 멈추고 물어라. 추측으로 큰 변경 금지.

## 6. 코드 규칙
- 컴포넌트는 작게 쪼개라. 섹션 단위로 분리.
- shadcn/ui 컴포넌트를 우선 쓰고 비슷한 걸 새로 만들지 마라.
- 폴더: `app/` 라우트, `components/` 공용 컴포넌트, `lib/` 유틸(Supabase 클라이언트 등).
- 이름은 명확하게. 줄임말 남발 금지.
- 주석은 "왜 이렇게 했는지"를 적어라.

## 7. 절대 하지 마라 (금지선)
- `.env`·비밀키·토큰을 코드에 직접 쓰지 마라. **환경변수로만.** 깃에 올리지 마라.
- Supabase 키 중 **service_role 키는 서버에서만** 쓴다. 클라이언트(브라우저)로 노출 금지. 공개 폼 제출은 anon 키 + RLS로 안전하게.
- 내가 시키지 않은 DB 마이그레이션을 함부로 돌리지 마라.
- 기존 파일을 통째로 갈아엎지 마라. 필요한 부분만 고쳐라.
- 임의로 라이브러리를 잔뜩 깔지 마라. 새로 깔 게 있으면 이유를 먼저 말해라. 특히 무거운 애니메이션 라이브러리(예: 무분별한 GSAP 도입) 지양.
- **"비교견적" 관련 기능·문구를 절대 넣지 마라.**
- **AI 생성 가짜 인테리어를 '실제 시공 사례'처럼 쓰지 마라.** (히어로의 분위기 이미지/영상은 가능, 포트폴리오는 실사진 자리만)
- PRD의 `[ ]`(숫자·실적)과 브랜드명은 **비워두거나 플레이스홀더로.** 지어내지 마라.

## 8. 글(카피) 규칙
- 화면 문구는 **PRD의 카피를 그대로** 쓴다. 임의로 바꾸지 마라.
- 새로 써야 하면 중학생도 이해할 수준, 짧은 문장, 어려운 한자어·전문용어 금지.
- 버튼은 동작으로: "무료 컨셉 상담받기". "확인 / 제출 / 더보기" 금지.

## 9. 디자인 규칙 (AvroKO 톤)
- 모바일 먼저. 작은 화면에서 안 깨지게.
- 배경: 따뜻한 아이보리/크림(순백 아님). 글자: 짙은 차콜. **포인트 색은 골드/브론즈 1개만**, 아주 절제.
- 폰트: 큰 제목·인용문 = 명조(세리프, 예: Noto Serif KR / 본명조). 메뉴·본문·버튼·작은 글씨 = Pretendard(산세리프). 본문 16px 이상. 크기 단계 3~4개로 통일.
- 사진은 풀블리드 대형. 여백 넉넉. 한 화면에 한 가지만 강조. 모서리 둥글기·그림자 전체 통일.
- 포트폴리오는 좌우 화살표(둥근 외곽선 버튼) + "01 / 08" 카운터 슬라이드 느낌.
- 스크롤 시 은은한 페이드인 정도만. 무거운 효과로 모바일 느려지면 안 됨.
- **CTA는 또렷하게:** "무료 상담" 버튼은 골드 또는 차콜로 채워 눈에 띄게 + 스크롤 따라다니는 고정 버튼. AvroKO식 약한 텍스트 링크로 숨기지 마라.
- 디자인 토큰(색·폰트·간격·둥글기)을 한 곳에 정의하고 전체에서 재사용.

## 10. 배포 전 점검
- 모바일에서 깨지는 곳 없는지
- 신청 폼 작동: Supabase 저장 + 운영자 알림 메일 + 완료 페이지 이동
- 폼 유효성 검사(연락처 형식 등)와 에러/성공 안내
- 공유 미리보기(제목·설명·대표이미지) 들어갔는지
- 오타·로딩 속도. 모든 페이지에 "무료 상담" 진입점이 있는지
